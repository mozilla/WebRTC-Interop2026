## Plan Improvement Suggestions

The following improvements would make this project easier to track over time:

1. **Add a WPT upstream tracking section.** The ~17 points from WPT test issues (iceConnectionState, handover, rid-manipulation, etc.) live as GitHub issues/PRs in the WPT repo, not Bugzilla. Future reports should include a table of WPT PR status (open, landed, or abandoned) alongside the Bugzilla section.

2. **Add a `first_seen_passing` velocity column to the test inventory.** Right now the inventory has no date column. Adding a "First Seen Passing" field (populated when a test first scores 100% in Firefox) will enable actual burndown velocity calculation. This field should be updated each time the report is regenerated.

3. **Automate score fetching.** The script at `resources/scripts/wpt_webrtc_scores.py` fetches live scores from the WPT API. Running it each cycle and diffing against the previous report's numbers would immediately highlight which tests changed. Consider adding it as a cron or CI step that posts a diff to the team Slack or bug.

4. **Track setParameters Glean telemetry explicitly.** Bug 1803388 / 2019370 (setParameters compat mode) is gated on a Glean telemetry check — the team needs to verify no deployed services still use the old API before the compat mode is disabled. This is an active prerequisite, not just a note. It should appear as a named milestone or blocker in the project tracker.

5. **Link Bug 1307994 (RTCIceTransport) to the meta bug.** It is currently absent from the meta bug's `depends_on` list despite controlling 4 test points. A missing link here means it won't appear in meta-bug dashboards or dependency sweeps.

6. **Owner assignments for unowned clusters.** RTCTransportStats (~23 pts), RTCRtcpMuxPolicy (14 pts), PRANSWER (8 pts), and setCodecPreferences (5 pts) are all unassigned. Staffing these is the single largest risk to project completion. Consider scheduling an explicit owner-assignment meeting before the end of Q2 2026.

7. **Add a per-report "Changes Since Last Report" delta.** Each new report should begin with a three-line summary: points added (new bugs/tests), points removed (fixed), and net change. This gives the team a quick pulse check without reading the full inventory.

8. **Consider encoding-test multi-browser failures separately.** `RTCRtpParameters-encodings` is failing in all three browsers at different rates (FF: 9/13, Chrome: 11/13, Safari: 6/13). Tests with multi-browser failures should be flagged as "spec under discussion" rather than "Firefox bugs" — otherwise the team may invest in a fix that just diverges Firefox from a spec change other browsers are already tracking.
